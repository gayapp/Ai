同趣 · 社区帖子（图文 / 视频）审核 — ai-guard 端实现方案

对接 App：app_79e81a7183747039 日期：2026-06-15 通道：moderate（/v1/moderate） 状态：设计已定稿，待开发

本文档是 docs/apps/同趣-社区帖子图文视频审核需求.md 的 ai-guard 端落地方案，覆盖三类发帖审核（纯文字帖 / 图文帖 / 视频帧帖），并按追加需求提供逐类结构化标签。

0. 背景与目标

同趣现有接入只覆盖 comment / nickname / bio / avatar，其中 avatar 是单张图片 URL。社区发帖审核分三类：

  • 纯文字帖 —— 只有标题/正文。
  • 图文帖 —— 标题/正文 + 1~N 张图片。
  • 视频帖 —— 同趣 VPS 先把视频间隔抽帧成 N 张帧图，再当"多图"送审（ai-guard 无需视频解码）。

目标产出：/v1/moderate 接受 biz_type='post' + image_urls[]，对一帖的文字与多图/多帧综合出单一 verdict，并额外返回 6+1 类结构化标签；支持 sync / auto（超时降级异步 + 回调）；后台完善到"枚举接入 + 多图查看 + 标签展示"。

铁律对齐：平台定位为成人男同社交 App 审核中间层 —— 合法 NSFW / 裸露 / 男同色情 绝不 reject，仅 CSAM / 广告引流 / 毒品 / 赌博 / 政治敏感 零容忍。

1. 关键设计决策

1.1 统一一个 biz_type='post' 覆盖三类

视频帧 = image_urls（需求 §2.3），所以"视频帖"和"图文帖"走完全相同的代码路径，仅靠 extra.post_type 区分（平台侧统计用，可选）。纯文字帖也用 post（image_urls 留空），不与 comment 合并 —— 帖子是顶层长内容，审核标准与可观测性应与评论分离，且让同趣一个 endpoint 形态搞定全部发帖审核。

1.2 视觉模型按"单次请求"动态决定

不按 biz_type 写死视觉开关，而是看请求是否带图：

请求形态模型说明
post + image_urls 为空（纯文字帖）grok-4-fast-non-reasoning文本模型，便宜快，可同步
post + image_urls 非空（图文 / 视频帖）grok-4（vision）多图一次性送入，综合判定

派生 helper：requestIsImage(biz, image_urls) = biz==='avatar' || (biz==='post' && image_urls?.length>0)。

1.3 结构化标签 labels（追加需求 · 重点）

图文/视频帖除 verdict 外，要像 IRC 端 Analyze 那样给出逐类标签：对每一类回答"是否有(detected) + 是什么(evidence)"。

做法：不另走 analyze 重型通道，而是在 moderate 输出上扩展一个可选 labels 数组，直接复用 Analyze 的 ViolationSchema 形状（{category, detected, confidence, evidence}，见 src/analyze/schema/media-analysis.ts L15-22）。这样同趣一次 sync 调用即拿到 verdict + 结构化标签，无需 analyze 的 queue / pull / ack / result_json 那套机制。

为什么不走 analyze：analyze 是队列驱动、pull/callback 交付、另一套重型契约，与同趣"默认同步拿结论"冲突；而 analyze 的 violations 与 face_coordinates 恰好就是本需求要的"是否有X + 是什么"，把这个形状借到 moderate 输出里即可，代价最小。

标签分类（6 类零容忍 + 1 类描述性）：

category含义对 verdict 的影响
minor_face疑似未成年人面孔出现detected 但无性化 → review（提示人工复核）
csam 零容忍未成年性化detected → reject
ad 零容忍广告引流detected → reject
drug 零容忍毒品detected → reject
gambling 零容忍赌博detected → reject
politics 零容忍政治敏感detected → reject
nsfw 描述性合法成人内容（裸露/男同色情）detected 可为 true，但不影响判定

2. 对外契约

2.1 请求 POST /v1/moderate

{
  "biz_type": "post",
  "biz_id": "01HXYZ_post_ulid",
  "content": "帖子标题\n帖子正文",
  "image_urls": [
    "https://f000.backblazeb2.com/file/duanvideo/temp/.../0.webp",
    "https://f000.backblazeb2.com/file/duanvideo/temp/.../1.webp"
  ],
  "user_id": "u_12345",
  "mode": "sync",
  "extra": { "post_type": "image", "media_count": 2 }
}

约束：image_urls 为 https:// 公网图，数组 ≤12；post 的 content 与 image_urls 至少其一非空；非 post 禁带 image_urls。

2.2 同步响应（沿用现结构 + 新增 labels）

{
  "request_id": "01HXYZ-...",
  "cached": false,
  "result": {
    "status": "reject",
    "risk_level": "high",
    "categories": ["porn", "ad"],
    "reason": "第2张图含未成年性化嫌疑，第3张图含微信引流",
    "labels": [
      { "category": "minor_face", "detected": true,  "confidence": 0.82, "evidence": "第2张图出现疑似未成年男性面孔" },
      { "category": "csam",       "detected": true,  "confidence": 0.71, "evidence": "第2张图未成年+裸露" },
      { "category": "ad",         "detected": true,  "confidence": 0.90, "evidence": "第3张图右下角微信号 abc123" },
      { "category": "drug",       "detected": false, "confidence": 0.0,  "evidence": "" },
      { "category": "gambling",   "detected": false, "confidence": 0.0,  "evidence": "" },
      { "category": "politics",   "detected": false, "confidence": 0.0,  "evidence": "" },
      { "category": "nsfw",       "detected": true,  "confidence": 0.95, "evidence": "成人男性裸露（合法，不影响判定）" }
    ]
  }
}

异步回调沿用 04-callback-spec.md 结构，同样追加可选 labels；biz_id / extra 原样回传；签名 X-Signature = hex(HMAC-SHA256(secret, raw_body))。

2.3 labels 字段定义

字段类型说明
categoryenumminor_face | csam | ad | drug | gambling | politics | nsfw
detectedboolean是否检出该类内容
confidencenumber 0~1置信度，默认 0
evidencestring ≤512"是什么" —— 命中位置/描述（中文，指明第几张图/帧）
兼容承诺（铁律 1）：biz_type 枚举加 post、新增可选 labels 均为只增不删；旧消费方（如一起看）忽略未知字段，不报错。改动须过 04-callback-spec.md 评审。

3. 端到端链路

1 POST /v1/moderate → HMAC 校验 + 限流
2 解析 ModerateRequestSchema（含 image_urls）→ 校验 biz_type 是否对该 app 启用
3 isImage = requestIsImage(...)；contentHash = computeContentHash(biz, content, image_urls)
4 前置漏斗：post 跳过 L1 低信噪，仍跑 L2 广告正则（caption）
5 dedup 命中 → 直接返回缓存结果
6 路由 post → grok；多图走 vision 模型，一次送入全部图
7 模型返回 → ModelOutput 校验（含 labels）→ 合并单 verdict
8 写库（含 image_urls / labels）+ 缓存；sync 直接返回 / auto 超时降级队列 + 回调

4. 代码改动清单

4.1 Schema / 契约 — src/moderation/schema.ts

  • BizType 枚举追加 "post"（只增不删）。
  • ModerateRequestSchema：新增 image_urls: z.array(z.string().url().startsWith("https://")).max(12).optional()；content 由 min(1) 放宽，用 .superRefine 保持各类型约束（非 post 必填且禁带 image_urls；post 二者至少其一）。
  • 新增 LabelCategory 枚举 + ModerationLabel（{category, detected, confidence, evidence}）。
  • ModelOutput / CallbackBody / 同步响应 / CachedResult 追加可选 labels。

4.2 多图贯穿管线

  • dedup.ts：computeContentHash(biz, content, imageUrls?) —— 仅当有图时按原序追加 "\n"+imageUrls.join("\n")；无图时与现状逐字节一致，不污染现有四类缓存。
  • router.ts：ProviderCallArgs 加 imageUrls?；DEFAULT_ROUTE 加 post: {primary:"grok", fallback:null}。
  • grok.ts：多图分支构造 [text?] + N×image_url + [text]，模型选 vision；新增含 labels 的 POST_STRUCTURE_SUFFIX。
  • pipeline.ts：ExecuteArgs / callProvider / ExecutionResult 透传 imageUrls 与 labels。

4.3 路由 / 异步 / 回调

  • routes/moderate.ts：isImage 改用 requestIsImage；hash 带 image_urls；willRequireCallback 覆盖 post；sync 响应带 labels。post 始终先试 sync，复用现有 PROVIDER_TIMEOUT && mode==='auto' 降级。
  • index.ts + types.ts：ModerationJob 加 image_urls?；队列消费同样透传；R2 证据保存收紧为仅 avatar（post 帧图不当头像存）。
  • callback/dispatcher.ts：组装 CallbackBody 时从库读回 labels，异步回调也含标签。

4.4 前置漏斗（修正一个正确性风险）— prefilter.ts

post 必须跳过 L1 低信噪 pass，否则"emoji 标题 + 违规图"会被误放行（图片才是风险载体）；但 保留 L2 广告正则跑在 caption 上。

4.5 Prompt 种子 — 新增 migration（只追加）

  • 新文件 migrations/00XX_seed_post_prompt.sql，为 (post, grok) 插入 v1 active prompt。
  • 遵循铁律 0（成人男同定位 + 多图综合判定 + 仅 6 类零容忍 + 视频帧语境）。
  • 必须产出 labels：逐类给 detected + evidence，并据此推导 verdict（任一零容忍 detected → reject；minor_face 无性化 → review；nsfw 不影响）。

5. 边界与约定（v1）

  • 不可达 / 非图 URL：v1 不做服务端可达性预检 —— 格式 + 数量 + https 校验通过即送 Grok；取不到图 → PROVIDER_ERROR → status=error。后续如需 unsupported_content 细分再迭代。
  • 单图 ≤8MB：同趣侧已压成 WebP w≤768，ai-guard 不强校验大小。
  • 多图 sync 偏慢：默认 SYNC_TIMEOUT_MS=10000；如常超时，上调 post 超时或引导用 mode=auto 降级。先用默认，观测后再调。
