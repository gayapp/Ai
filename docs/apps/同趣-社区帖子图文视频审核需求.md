## 1. 背景与诉求

同趣社区现要对**用户发帖**做审核，分两类：

- **图片帖**：1~N 张图片 + 标题/正文。
- **视频帖**：同趣 VPS（moderation-service）会先对视频**间隔抽帧**得到 N 张帧图，再当作"多图"送审。

当前 ai-guard 对同趣只启用了 `comment / nickname / bio / avatar`，其中 **avatar 是单张图片 URL**。社区帖需要 **`biz_type='post'` + 多图 `image_urls` 数组**，这是现有契约没有的，需要 ai-guard 端开发。

> 成本背景：同趣侧已在 VPS 把图片**压缩成小图（WebP，w≤768）**后再送审，URL 指向 B2 公网图。ai-guard 只需按 URL 拉图分析。

## 2. 需求清单（ai-guard 端要做的）

### 2.1 新增 `biz_type = 'post'`
- 为同趣 App `app_79e81a7183747039` 启用 `post`。
- `/v1/moderate` 接受 `biz_type='post'`。
- 回调 schema 的 `biz_type` 枚举增加 `post`（[04-callback-spec.md](../04-callback-spec.md) §字段定义，向后兼容新增枚举值）。

### 2.2 新增多图入参 `image_urls: string[]`
- `/v1/moderate` 请求体支持 `image_urls`（字符串数组，每个 `https://` 公网图 URL）。
- 仅 `biz_type='post'`（及未来需要的多图类型）使用；avatar 仍可用单 `content`=URL 不变。
- 约束建议：数组长度上限（如 ≤12）、单图 ≤8MB、非图/不可达按 `unsupported_content` 处理。
- **整组出一个 verdict**：对一帖的多张图（或一视频的多帧）综合判定，返回单一 `status`（命中任一严重违规即 reject）。`categories` 合并去重。

### 2.3 视频帧审核（复用多图能力）
- 视频帖的多帧图就是 `image_urls`，**无需视频解码能力**，ai-guard 当多图处理即可。
- `extra` 会带 `{ "post_type": "video", ... }` 供平台侧统计/调 prompt（可选用）。

### 2.4 视觉模型策略
- post 图片/帧需视觉模型（同 avatar 走 Gemini；Grok 无视觉）。请确保 `post` 的 provider 策略落到支持视觉的模型，多图一次性送入。
- 若多图导致单次 token 过大，平台可自定义分批/采样策略，但对调用方仍是**一次请求、一个结论**。

### 2.5 同步 + 异步 + 回调
- 支持 `mode=sync`（同趣默认同步拿结论）与 `mode=auto`（超时降级异步 + 回调）。
- 回调契约沿用现有（[04-callback-spec.md](../04-callback-spec.md)）：`X-Signature = hex(HMAC-SHA256(secret, raw_body))`，`biz_id` 原样回传（= 同趣 `post_id`），`extra` 原样回传。

## 3. 接口契约（期望）

### 3.1 请求 `POST /v1/moderate`
```json
{
  "biz_type": "post",
  "biz_id": "01HXYZ_post_ulid",
  "content": "帖子标题\n帖子正文",
  "image_urls": [
    "https://f000.backblazeb2.com/file/duanvideo/temp/2026-06-15/post/01HXYZ/0.webp",
    "https://f000.backblazeb2.com/file/duanvideo/temp/2026-06-15/post/01HXYZ/1.webp"
  ],
  "user_id": "u_12345",
  "mode": "sync",
  "extra": { "post_type": "image", "media_count": 2 }
}
```

### 3.2 同步响应（期望，沿用现有结构）
```json
{
  "request_id": "01HXYZ-...",
  "cached": false,
  "result": {
    "status": "pass | reject | review",
    "risk_level": "safe | low | medium | high",
    "categories": ["porn", "..."],
    "reason": "一句中文说明"
  }
}
```

### 3.3 回调（异步时，沿用现有 schema，仅 biz_type 多一枚举）
```json
{
  "schema_version": "1.0",
  "request_id": "01HXYZ-...",
  "app_id": "app_79e81a7183747039",
  "biz_type": "post",
  "biz_id": "01HXYZ_post_ulid",
  "user_id": "u_12345",
  "status": "pass | reject | review | error",
  "risk_level": "safe|low|medium|high|null",
  "categories": ["..."],
  "reason": "...",
  "extra": { "post_type": "image", "media_count": 2 },
  "created_at": "2026-06-15T08:00:00Z"
}
```

## 4. 调用方信息

- 调用方：同趣 **VPS（moderation-service，物理机）**，Python `httpx`，HMAC-SHA256 签名（按 `同趣-integration.md` §3）。
- 凭据：复用同趣 App `app_79e81a7183747039` 的 secret（无需新发 App）。
- 量级：社区发帖触发，峰值受同趣 100 QPS 限额约束；存量老帖回填会有短时批量（同趣侧会限速）。
- `biz_id` = 同趣 `post_id`（ULID 字符串，≤128）。
