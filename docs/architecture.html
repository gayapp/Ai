1. 概览

ai-guard 统一为公司各 C 端应用提供 UGC 审核能力。应用只对接一个 API，结果以固定 JSON 格式返回或回调。内部上游对接 Grok（文本）和 Gemini Vision（图片），支持熔断切换、去重缓存、prompt 热更新、审计统计。

业务类型
comment · nickname · bio · avatar
响应模式
混合自适应（sync / async / auto）
鉴权
HMAC-SHA256 + Nonce + Timestamp
计费
不做（仅统计：请求数/token/命中率）
模型
grok-4-fast-non-reasoning / gemini-2.5-flash
延迟
文本 ~500ms · 图片 ~3.5s · 缓存 ~400ms

8. 指令（Prompt）存哪里、怎么改

8.1 存储位置

所有 prompt 存在 D1 的 prompts 表，按 (biz_type, provider, version) 唯一。一个 (biz_type, provider) 同时只有一个 is_active=1。

CREATE TABLE prompts (
  id         INTEGER PRIMARY KEY,
  biz_type   TEXT,          -- comment | nickname | bio | avatar
  provider   TEXT,          -- grok | gemini
  version    INTEGER,
  content    TEXT,          -- 整段 prompt 正文
  is_active  INTEGER,
  created_by TEXT,
  created_at INTEGER,
  UNIQUE(biz_type, provider, version)
);

8.2 初始默认指令（已 seed 入库）

本次初始化通过 migration 0001 + 0002 直接写库，共 14 条（每个 biz_type × 每个 provider × v1（保留）+ v2（激活））。

v2 默认 prompt 基于"only visible evidence"、"灰色地带给 review、不一刀切"的原则编写，结构参考本项目 C:\code\irc\python 的 LLM 审核范式，但内容已改写为通用 UGC 场景（不含成人内容平台专属规则）。

每条 prompt 都声明了以下判定级别：

  • reject + risk_level=high：色情低俗 / 人身攻击 / 政治敏感 / 广告引流 / 刷屏 / 暴力威胁
  • reject + risk_level=medium：引战歧视 / 软色情暗示 / 隐藏广告
  • review：意思含糊但疑似违规、谐音嫌疑、幽默/反讽不确定
  • pass + risk_level=safe：正常表达

8.3 热更新（生产环境）

不用改代码、不用重新部署。通过 Admin API：

# 发布新版本（同 biz_type+provider 下 version 自动 +1，旧版本 is_active 置 0）
curl -X POST https://ai-guard.schetkovvlad.workers.dev/admin/prompts \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "content-type: application/json" \
  -d '{
    "biz_type": "comment",
    "provider": "grok",
    "content": "<你的新 prompt 正文>",
    "created_by": "alice"
  }'

发布后：

  • D1 写入新 version 记录，旧 active 自动失活
  • KV PROMPTS 缓存失效（60s 内全边缘生效）
  • KV DEDUP_CACHE 因 key 含 prompt_version 自然失效，无需手动清理

8.4 查看 / 回滚 / 干跑

# 列全部版本
GET /admin/prompts?biz_type=comment&provider=grok

# 回滚到旧版本（把目标 id 置 is_active=1，其他置 0）
POST /admin/prompts/:id/rollback

# 干跑：用新 prompt 跑 N 条样本，看输出是否合 schema
POST /admin/prompts/dry-run
{ "biz_type":"comment","provider":"grok","content":"...","samples":["样本1","样本2"] }
纪律：prompt 中无论写什么，都不可能破坏回调 JSON 结构——结构由代码层 Zod schema 锁定（src/moderation/schema.ts），模型返回不合规直接标 status=error。所以运营可以放心调 prompt 的判断规则，不会泄漏异常字段到业务侧。

9. 数据可见性（全部可查）

所有审核记录、回调投递、统计指标均可通过 Admin API 查询。Bearer token 鉴权（prod 和 dev 各独立 token）。

9.1 可查哪些数据？

数据接口说明
所有审核请求列表（分页） GET /admin/stats/requests?limit=N&app_id=&biz_type=&status=&from=&to= 支持按 app / 业务 / 状态 / 时间筛选
单条审核详情（含 token、延迟、prompt_version、extra） GET /admin/stats/requests/:id 完整字段，含用户原文 biz_id / 自定义 extra
汇总统计（通过率 / token / 缓存命中率 / 状态分布） GET /admin/stats/summary?from=&to=&app_id= 可按 app 过滤
被拒用户 Top N（反滥用） GET /admin/stats/top-users?app_id=&limit=20 按 reject 次数排序
回调投递记录 GET /admin/stats/callbacks?limit=50&failed=1 每次投递的 http 状态 / 重试次数 / 最后错误
应用列表 / 详情 GET /admin/apps[/:id] 不返回 secret（只能查看），只能通过 rotate-secret 接口换发
Prompt 全部版本（含已回滚的） GET /admin/prompts?biz_type=&provider= 每个 biz_type × provider 的全版本列表 + content
